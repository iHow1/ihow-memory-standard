## Purpose / 目的

This document describes what changes between iHow Memory Standard v0.1 (published) and the upcoming v0.2. It is **incremental**: every section in v0.1 that this delta does not mention remains in force unchanged.

本文档描述 iHow Memory Standard 从已发布的 v0.1 到即将发布的 v0.2 的**增量变更**。v0.1 中本文档未提及的部分继续生效。

v0.2 stays inside the same narrow promise as v0.1: agents should be able to share durable, scoped, auditable project memory without depending on hidden chat history. v0.2 does **not** define storage engines, hosted services, model providers, or commercial certification. What it adds is normative semantics for evaluating whether an implementation, runner, or external runtime actually exercises that promise.

v0.2 仍然延续 v0.1 的窄口径承诺：让 agent 能共享持久、可限定、可审计的项目记忆，不依赖隐藏会话历史。v0.2 **不**定义存储引擎、托管服务、模型供应方或商业认证。它只补充评估"实现 / runner / 外部运行时是否真的兑现承诺"所需的规范化语义。

## 0. Recap — what v0.1 already provides / v0.1 已经提供的内容

The following pieces from v0.1 are **not changed** by this delta and remain authoritative:

- 5 reliability scenarios (S1 Cross-Tool Handoff, S2 Feedback Pattern Capture, S3 Constraint Preservation, S4 Human Team Handoff, S5 Model Migration) with Given / When / Then / 失败模式 / 验收标准.
- Protocol draft with 4 interface semantics (`events`, `context`, `writeback`, `audit`) and JSON shape sketches.
- 4-layer namespace (`tenant` / `customer` / `project` / `user`) and boundary rules.
- Safety boundaries on what must not be stored as ordinary memory.
- Compliance test preview (informative list of behaviors a future suite should test).
- Non-Goals (no storage engine, no hosted service, no specific provider, no tool integration code, no private operations standardization).

v0.2 builds on top of those without re-litigating them.

## 1. What is new in v0.2 / v0.2 的新增内容

The delta introduces five additions and one clarification:

- **Section A. Runner result classification** (normative): formalizes the four result levels `PASS` / `PARTIAL` / `NOT_APPLICABLE` / `BLOCKED` and how per-check results aggregate.
- **Section B. Runner applicability matrix** (normative): describes which scenarios are reasonable to evaluate for which runner architecture types.
- **Section C. Conformance test points per scenario** (normative): names a small set of stable, atomic test points per scenario so that fixtures from different runners are comparable.
- **Section D. Fixture policy** (normative): elevates the three policy flags already enforced by `fixture-runner.mjs` to spec-level requirements and adds a `references_only_pinned` rule.
- **Section E. Self-conformance vs. experimental external runner** (normative clarification): names the boundary between an implementation evaluating its own behavior and an experimental runner evaluating a third-party runtime, and the criteria for promoting an experimental runner to `official`.
- **Section F. Out of scope for v0.2** (informative): carries forward v0.1 non-goals and adds explicit non-goals introduced during W6 work.

Section G is an informative fixture schema hint that does **not** become a normative API; Section H records open questions for v0.3.

## A. Runner result classification / 结果分级 (normative, NEW)

A conformance runner emits a result **per scenario per runtime**. The result must be one of four levels:

- `PASS` — the runtime's synthetic fixture demonstrates all required behaviors for the scenario.
- `PARTIAL` — the runtime exhibits some required behaviors and explicitly names gaps for the rest.
- `NOT_APPLICABLE` — the runtime's architecture does not expose the capability that the scenario requires (e.g., a black-box export/import runtime has no native writeback API to evaluate).
- `BLOCKED` — the runner cannot evaluate because the environment, fixture, or required input is unavailable.

A runner MUST also report **per-check** results. Each scenario carries an ordered list of named atomic checks; each check has a boolean `ok`. The scenario-level status is derived from the per-check results plus optional explicit gaps.

每个 scenario × runtime 必须输出四级结果之一；每条 scenario 必须给出有序的 check 列表，每个 check 有 `ok: true|false`。scenario 等级由 check 列表 + 显式 gaps 推导。

### A.1 Per-scenario derivation rules / 单 scenario 推导规则

A runner emitting a scenario result MUST satisfy the following:

- `PASS` requires zero failed checks.
- `PARTIAL` requires at least one passed check AND at least one named gap (a failed check, or an entry in `gaps[]` explaining what is missing).
- `NOT_APPLICABLE` requires zero checks AND at least one explicit gap explaining why the scenario is structurally not evaluable for this runtime.
- `BLOCKED` requires at least one explicit gap describing what is missing (fixture, environment, dependency) AND should be treated as a runner-side failure to evaluate, not a runtime-side absence of capability.

A runner that emits `PASS` while a check has `ok: false` MUST be rejected at fixture validation time.

### A.2 Per-runtime aggregation rules / 单 runtime 汇总规则

When a runtime has multiple scenarios, the runner aggregates them into a single `overall_status` for reporting:

- If any scenario is `BLOCKED`, the runtime is `BLOCKED`.
- Else if all scenarios are `PASS`, the runtime is `PASS`.
- Else if all scenarios are `NOT_APPLICABLE`, the runtime is `NOT_APPLICABLE` (this is rare; usually means the runner is not picking up scenarios for this runtime).
- Else, the runtime is `PARTIAL`.

`PARTIAL` is the expected steady-state for most external runtimes in v0.2.

### A.3 What classification does NOT mean / 分级不代表的事

- `PASS` against an experimental runner fixture is **not** an iHow Memory certification. v0.2 does not issue certifications.
- `PARTIAL` is **not** a failure verdict. It is the honest description of an architecture that partially covers the spec.
- `NOT_APPLICABLE` is **not** a way to silently hide a gap. The runner MUST explain why the capability is absent.
- `BLOCKED` is a runner-side problem, not a runtime-side problem. A runtime is never `BLOCKED` because of its own behavior; only because the runner cannot reach it.

## B. Runner applicability matrix / 适用性矩阵 (normative, NEW)

v0.2 categorizes external runtimes by **architecture type**, not by brand. A runner SHOULD self-classify before claiming a scenario status. The matrix below names which scenarios are reasonable to evaluate (`E`), reasonably `NOT_APPLICABLE` by design (`N`), or context-dependent (`C`) for each architecture type.

| Architecture type | S1 Cross-Tool Handoff | S2 Feedback Pattern | S3 Constraint Preservation | S4 Human Team Handoff | S5 Model Migration |
| --- | --- | --- | --- | --- | --- |
| **Local-first protocol-aligned runtime** (self) | E | E | E | E | E |
| **Markdown / file-based handoff** | E | C | N | E | C |
| **Black-box export / import** | E | N | N | C | E |
| **Source-trace export / import** | E | E | C | C | E |
| **Partial-interface / mixed-capability runtime** (covers some but not all of the four protocol interfaces) | C | C | C | C | C |

Legend / 说明:

- `E` — Eligible to attempt; runner can produce `PASS` or `PARTIAL` based on fixture evidence.
- `N` — Default `NOT_APPLICABLE` is reasonable; runner SHOULD explain the architectural reason.
- `C` — Context-dependent; runner MUST decide per fixture and document its choice.

The matrix is **descriptive**, not exhaustive. A new architecture type may appear in v0.3.

矩阵给出**默认期望**，不是封闭枚举。新的架构类型可以在 v0.3 里追加。

### B.1 Worked example / 范例对照

The experimental runners shipped with `c3728d1` self-classify into the matrix as follows (informative, not normative; each runner picks its own architecture row):

- A runner whose runtime exposes a partial set of the protocol interfaces (and does not assert full alignment) self-classifies as **partial-interface / mixed-capability** and emits per-scenario `PARTIAL` / `NOT_APPLICABLE` as appropriate.
- A runner whose runtime exports / imports memory state without exposing the full protocol surface self-classifies as **black-box export / import** and is allowed to mark scenarios that require writeback semantics as `NOT_APPLICABLE`.
- A runner whose runtime is built on a markdown / file-based handoff package self-classifies as **markdown / file-based handoff** and is allowed to attempt the human-team-handoff scenario directly.
- A runner whose runtime is centred on source-trace export / import self-classifies as **source-trace export / import** and is allowed to attempt model-migration and feedback-trace scenarios.

The point of this list is the **self-classification pattern**, not who picks which row. Specific runner verdicts (which scenarios each shipped fixture emits as `PASS` / `PARTIAL` / `NOT_APPLICABLE`) live in each runner's own README and fixture file, not in this spec.

## C. Conformance test points per scenario / 每个场景的一致性测试点 (normative for `spec_version: v0.2` fixtures)

v0.2 introduces a per-scenario list of atomic test points. The goal is to make fixtures from different runners comparable when they choose to declare v0.2.

**Compatibility rule** (this rule precedes §I): a fixture that does NOT declare `spec_version: v0.2` is evaluated as a v0.1 fixture and is **not required** to use the names below. A fixture that declares `spec_version: v0.2` SHOULD use the core check names below for any scenario it attempts, MAY add runner-specific checks, and MAY skip any check that is not part of the scenario's **core** subset (marked with `★`).

The test points are deliberately small and behavior-shaped; they do not prescribe an implementation.

### C.1 Scenario 1 — Cross-Tool Handoff / 跨工具接力

- `T1.1` ★ Handoff package contains at least one completed item.
- `T1.2` ★ Handoff package contains at least one open item.
- `T1.3` Handoff package contains at least one preserved constraint.
- `T1.4` The next-action statement is consistent with the recorded checkpoint.
- `T1.5` The package is bounded; it is not the full raw history.
- `T1.6` Unrelated project / user context is absent from the package.
- `T1.7` Context selection metadata explains why each item is included. *(SHOULD only; depends on `context` interface capabilities from protocol-draft-v0.1.)*

### C.2 Scenario 2 — Feedback Pattern Capture / 反馈规律沉淀

- `T2.1` ★ A repeated feedback signal is converted into one rule (not stored as N isolated copies).
- `T2.2` ★ The rule carries `scope`, `source_event_ids`, `confidence`, and `review_status`.
- `T2.3` A later task retrieves and demonstrates the rule.
- `T2.4` The rule's source events are traceable.
- `T2.5` The rule supports revise / disable / scope-narrow actions.

### C.3 Scenario 3 — Constraint Preservation / 禁忌约束执行

- `T3.1` ★ The constraint is stored with `priority`, `scope`, `source`, and `status`.
- `T3.2` ★ The constraint is retrieved before relevant generation or action.
- `T3.3` A deliberate conflict between an instruction and the constraint is surfaced (block / warn / confirm depending on severity).
- `T3.4` Hard constraints and soft preferences remain distinguishable.
- `T3.5` The constraint survives session, tool, and agent switch within the same project.

### C.4 Scenario 4 — Human Team Handoff / 新人接手

- `T4.1` ★ Onboarding package separates **facts / decisions / assumptions / blockers / next-steps** as named sections.
- `T4.2` ★ Each important claim links to its source event or artifact.
- `T4.3` The package is short enough for a new operator to read quickly (a runner MAY define a soft length bound for its fixture; the spec does not pin a numeric value in v0.2).
- `T4.4` Uncertainty is presented separately from confirmed facts.
- `T4.5` Inactive / superseded artifacts are excluded from the active package.

### C.5 Scenario 5 — Model Migration / 跨模型迁移

- `T5.1` ★ The same project state is retrievable by two different model-facing clients.
- `T5.2` ★ Key facts, decisions, and constraints survive the migration.
- `T5.3` The context package is bounded to the receiving model's practical limits.
- `T5.4` Source traceability survives the migration.
- `T5.5` The stored format is model-neutral (not prompt-templated for one provider).

**Core subset (`★`)**: the 10 checks marked `★` above are the minimum behaviors a v0.2-declared fixture SHOULD attempt for any scenario it includes. The other 17 checks are recommended; a fixture MAY skip them when the runtime architecture does not exercise the underlying behavior.

### C.6 Naming convention for additional checks / 自定义检查命名约定

Runner-specific additional checks SHOULD use names that:

- start with the lowercase scenario tag (`s1.`, `s2.`, ..., `s5.`) when domain-specific;
- avoid claiming behaviors the runner cannot verify;
- avoid embedding tenant / customer / project / user content.

The spec does not enforce these naming hints; they exist so that aggregated runner reports remain readable.

## D. Fixture policy / fixture 策略 (normative, formalized + NEW)

`fixture-runner.mjs` already enforces three policy flags. v0.2 elevates them to spec-level requirements and adds one new flag.

### D.1 Required policy flags / 必填策略字段

A fixture submitted to a v0.2-aware runner MUST set:

- `policy.public_fixture === true` — the fixture content is safe to publish in a public spec repo. No private memory, no customer-identifying content, no tokens / keys / credentials.
- `policy.uses_synthetic_data === true` — fixture content is constructed for the test; it is not a copy of real customer or user memory.
- `policy.no_official_conformance_claim === true` — the fixture is part of an experimental runner unless explicitly promoted; running it does **not** assert that the third-party runtime conforms to iHow Memory.

A fixture that omits or sets any of these flags to false MUST be rejected at fixture validation time.

### D.2 New v0.2 requirement: scoped reference pinning / 引用钉版分级 (NEW)

A fixture that names an external runtime MUST include a non-empty `references` array. The strictness of pinning depends on the runner's `classification`:

- **Official runner** (`classification: "official"`): MUST pin at least one of the following: a commit SHA, a release version tag, or a PR number plus head commit at the time of evaluation. Free-text-only references are not accepted.
- **Experimental runner** (`classification: "experimental"`, default): SHOULD pin a commit / version / PR + head commit when feasible; MAY use a dated free-text reference of the form `"draft snapshot 2026-05-22; no upstream endorsement implied"`. The date is required so future readers can locate the rough behavior the fixture targeted.

A fixture that mentions an external runtime by name with an empty `references` array, or with a reference that is neither a pinned artifact nor a dated free-text snapshot, MUST be treated as a runner-side configuration error (BLOCKED).

Rationale: pinning is needed for reproducibility, but experimental runners often probe a moving upstream where pinning every commit creates excessive maintenance load. The dated snapshot form preserves auditability without forcing experimental fixtures to track upstream commits one-to-one.

### D.3 Optional policy fields / 可选字段

A fixture MAY set:

- `classification: "experimental" | "official"` — default is `experimental`. See Section E for promotion criteria.
- `references[].note` — short reason this reference exists (e.g., "PR head recorded as experimental reference only").
- `policy.notes` — free-form explanation for reviewers; never used by the runner for verdict logic.

## E. Self-conformance vs. experimental external runner / 自我一致性 vs. 实验性外部 runner (normative clarification)

A v0.2-aware runner runs in one of two modes. The boundary between them is not stylistic; it changes what the runner is allowed to claim.

### E.1 Self-conformance / 自我一致性

A self-conformance runner evaluates the behavior of the implementation that owns the runner. It MAY:

- emit `PASS` on a scenario it actually exercises.
- claim that the owning implementation passes specific scenarios at a specific commit.
- be referenced as evidence of internal correctness.

A self-conformance runner MUST NOT:

- evaluate third-party runtimes inside the same fixture set.
- claim that a third-party runtime conforms to the spec.

The published `ihow-memory-core` runner under `iHow1/ihow-memory-core/conformance/runners/ihow-memory/` is the example self-conformance runner for v0.2.

### E.2 Experimental external runner / 实验性外部 runner

An experimental external runner evaluates a third-party runtime's behavior using synthetic fixtures. It MAY:

- emit `PASS` / `PARTIAL` / `NOT_APPLICABLE` per scenario based on the synthetic fixture.
- describe what behavior the fixture exercises.
- record references to upstream PRs / commits / versions.

An experimental external runner MUST NOT:

- claim that the third-party runtime "conforms to iHow Memory protocol".
- imply upstream endorsement or certification.
- ingest private memory or customer data into the fixture.
- modify the third-party runtime's source repo in any way as part of running the conformance suite.

The shipped runners (`openclaw`, `openviking`, `gbrain`, `m-flow`) are all experimental external runners in v0.2.

### E.3 Promotion criteria — experimental → official / 实验性 → 正式认定的最小条件

A runner moves from `experimental` to `official` only when **all** of the following are true:

- The third-party runtime has merged the referenced behavior or artifact described in the fixture (no longer a draft PR or unmerged branch).
- The fixture's pinned reference is updated to a post-merge commit on the upstream default branch.
- The maintainers of the third-party runtime have explicitly acknowledged the runner exists and have not objected to the `official` classification, OR the runner's runtime is the standard's own self-conformance runtime.
- The maintainers / stewards of the `ihow-memory-standard` repo approve the promotion in writing (issue, PR review, or governance record). Governance role definitions live in `CONTRIBUTING.md` and `GOVERNANCE.md`, not in this spec.

Promotion is **per runtime**, not per scenario. A runtime that meets E.3 may still emit `PARTIAL` or `NOT_APPLICABLE` for some scenarios. Promotion changes the `classification` field, not the per-scenario verdict.

**Downgrade and removal** (NEW in v0.2): the maintainers of a third-party runtime MAY at any time request that an `official` runner be downgraded back to `experimental`, or removed from the standard repo entirely. Such requests SHOULD be honored within a reasonable time window. Standard repo maintainers MAY also unilaterally downgrade a runner if the upstream runtime behavior diverges enough that the existing fixture is misleading.

## F. Out of scope for v0.2 / v0.2 非目标 (informative)

v0.2 deliberately does **not** define any of the following:

- A normative storage schema or persistence engine.
- A hosted SaaS endpoint or wire protocol.
- A specific model provider, embedding model, or vector store.
- A required tool integration (IDE, chat client, runner harness).
- A commercial certification, badge, or paid compliance program.
- A required event log file format. (`memory/_events/<date>.ndjson` is one adapter convention, not a normative wire format.)
- A required review queue UI or human-in-the-loop UX.
- Quantitative thresholds for "enough evidence" in S2 pattern recognition (left to v0.3 design).
- An aggregation rule for multiple runners against the same runtime (a follow-up RFC will address this).

These remain non-goals in v0.2 to keep the standard small and easy to audit.

## G. Implementation hint — fixture schema (informative, not normative)

This section is **informative**: it documents the JSON shape that the current `fixture-runner.mjs` validates. It is not a normative interface; it exists so that early adopters can write fixtures without reading runner source code.

```json
{
  "runtime": "example",
  "label": "Short human description",
  "classification": "experimental",
  "references": [
    "version:0.1.0 OR headRefOid:abc123... OR draft note"
  ],
  "policy": {
    "public_fixture": true,
    "uses_synthetic_data": true,
    "no_official_conformance_claim": true
  },
  "scenarios": [
    {
      "id": "S1",
      "title": "Cross-Tool Handoff",
      "status": "PARTIAL",
      "checks": [
        { "name": "T1.1 handoff contains completed item", "ok": true },
        { "name": "T1.2 handoff contains open item", "ok": true },
        { "name": "T1.7 selection metadata present", "ok": false }
      ],
      "evidence": [
        "Synthetic export/import package shape ..."
      ],
      "gaps": [
        "Bounded context selection metadata not produced by this runner."
      ]
    }
  ]
}
```

Field semantics:

- `runtime`, `label` — required strings.
- `classification` — defaults to `experimental` when omitted.
- `references` — required for external runtimes; see Section D.2.
- `policy` — three required flags; see Section D.1.
- `scenarios[].id` and `.title` — required strings; SHOULD use S1..S5 ids from `reliability-scenarios-v0.1.md` plus optional vendor-specific scenarios.
- `scenarios[].status` — one of `PASS` / `PARTIAL` / `NOT_APPLICABLE` / `BLOCKED`.
- `scenarios[].checks[]` — ordered list; each has `name` and `ok`.
- `scenarios[].evidence` — short factual notes about what the fixture demonstrates.
- `scenarios[].gaps` — explicit human-readable gaps; required when status is `PARTIAL` / `NOT_APPLICABLE` / `BLOCKED`.

## H. Open questions for v0.3 / 留给 v0.3 的开放问题

The following questions are explicitly **deferred** to v0.3 design work and are appropriate topics for public RFC discussion. Other design questions remain open inside the standard repo's internal design notes; only the questions below are public.

- How should v0.3 handle nested namespaces (e.g., sub-projects inside a project)?
- How should v0.3 aggregate results across multiple independent runners evaluating the same runtime?
- Should v0.3 introduce a normative `audit.explain_excluded` shape for "why was this memory item NOT included in the context package"?

## I. Versioning and compatibility / 版本与兼容性

v0.2 is **strictly additive** with respect to v0.1:

- A v0.1 fixture that passed `fixture-runner.mjs` at commit `c3728d1` continues to pass under v0.2-aware runners.
- A v0.1 self-conformance runner that produced `5/5 PASS` against the v0.1 scenarios continues to produce `5/5 PASS` under v0.2 (the scenarios themselves did not change).
- A v0.2-aware runner SHOULD additionally enforce Sections D.1, D.2, and the per-scenario test point names from Section C when a fixture declares conformance to v0.2.

No v0.1-shipped behavior is deprecated by this delta.
