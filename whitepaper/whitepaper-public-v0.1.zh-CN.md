# iHow Memory · 白皮书 v0.1

**多 Agent 工作流的本地优先记忆与交接可靠性层**
*Local-First Memory & Handoff Reliability Layer for Multi-Agent AI Workflows*

---

## 摘要 / Executive Summary

AI Agent 在单次对话里已经足够聪明，但跨会话、跨工具、跨人接手之后会**集体失忆**。客户偏好、改稿规律、关键决策、交接边界——这些项目级状态被困在某个聊天历史里，下一次工作必须从头重新讲一遍。

**iHow Memory 解决这个问题**。它是一套面向多 Agent 工作流的**本地优先记忆与交接可靠性层**：用文件协议沉淀项目状态、用通用接口对接任意 AI 工具、用可审计的方式管理人审与回滚、用一套公开验收套件证明系统真的"不失忆"。

我们已经在国际公开评测 **LongMemEval Public-S** 全部 470 个样本上达到 **recall_all@10 = 1.000**，NDCG@10 = 0.9457，长期记忆复杂任务测试集全部通过，并且已在三个真实 AI 工具之间完成跨工具接力闭环。

本白皮书阐述 iHow Memory 的问题陈述、设计原则、5 个核心可靠性场景、协议草案、验证证据，以及面向社区的开放标准倡议。

## 1. 问题陈述 / The Problem

### 1.1 现状：AI 在单点聪明，在长链失忆

过去两年 AI Agent 工具爆发——编程助手、内容生成、客户支持、研究分析全部 AI 化。当一个任务在**单次对话**里完成时，AI 表现可以接近专家。

但**真实工作不是单次对话**。一个客户项目持续数周，一段开发任务横跨数月，一份调研要在多个工具之间接力。一旦工作流跨越 session、跨越工具、跨越人，AI 的"聪明"就退化成"健忘"——它每次都从零开始认识你。

### 1.2 三种典型失忆

| 失忆类型 | 真实代价 |
|---|---|
| **改稿失忆 / Revision amnesia** | 客户反馈过 3 次的偏好，AI 第 4 次还会犯同样错误 |
| **工具失忆 / Tool-swap amnesia** | 从一个 AI 工具切到另一个 AI 工具，项目状态、技术决策、风险记录全部清零 |
| **接手失忆 / Handoff amnesia** | 新人接手客户项目，要从聊天记录里翻找过去半年的上下文，效率从第一天起就是 -70% |

### 1.3 现有方案为什么不够

市面上主要的记忆方案，按局限分类：

- **向量数据库黑盒**：能"想起"事实，但人不可读、不可审、不可质疑。一旦出错没办法回滚到具体源头。
- **云端 SaaS 记忆**：方便但要把客户敏感资料上传给第三方。金融、医疗、法律、政企不可行。
- **单 Agent 记忆模块**：每个 AI 工具自带一套记忆，互相不互通。换工具等于重新建模。
- **手工文档 + 知识库**：完全依赖个人维护，长期不可扩展。

### 1.4 真正的瓶颈是"记忆基础设施"

当企业 / 团队 / 个人**从单 Agent 试验进入多 Agent 流程**，记忆问题从"feature"升级到"control plane"——它直接决定 AI 自动化能不能从"一次性脚本"变成"可持续运营资产"。

iHow Memory 就是为解决这个 control plane 问题而设计。

## 2. iHow Memory 定位 / Positioning

### 2.1 一句话定义

> **iHow Memory 是面向多 Agent 工作流的本地优先记忆与交接可靠性层。**
>
> *iHow Memory is the local-first memory and handoff reliability layer for multi-agent AI workflows.*

### 2.2 它不是什么

- 它**不是**通用对话记忆（如 Mem0 风格的"AI 记住用户喜好"）
- 它**不是**向量检索数据库的替代品
- 它**不是**单 Agent 框架（如 Letta / MemGPT 风格）
- 它**不是**云 SaaS 服务

### 2.3 它是什么

- 它是项目级、文件协议、可审计、跨 AI 工具的记忆中台
- 它是多 Agent 之间稳定接力的可靠性层
- 它是企业可以私有化部署、不需要让数据离开本地的基础设施
- 它是公开的协议规范 + 公开的验收测试，**邀请社区共同定义"什么叫合格的 AI 记忆"**

### 2.4 适用场景

- 内容生产团队（短视频、营销、品牌内容）
- AI 编程与长期开发项目
- 企业内部 AI 助手与知识沉淀
- 咨询 / 设计 / 交付项目交接
- 研究调研与资料长期管理
- 高敏数据私有部署（金融、医疗、法律、政企）

## 3. 核心设计原则 / Core Design Principles

### 3.1 本地优先 / Local-First

记忆数据默认存储在用户或客户自己的环境里。云端是可选项，不是默认值。这让 iHow Memory 天然适合高敏行业。

### 3.2 文件协议 / File Protocol

核心记忆使用 Markdown + YAML frontmatter 存储。人可以直接打开、阅读、修改、回滚。**不是向量黑盒，是可审计的真相源**。

### 3.3 模型无关 / Model-Agnostic

iHow Memory 不绑定任何 LLM。无论后端是主流商业模型、国内模型，还是自部署的开源模型，记忆层的接口一致。

### 3.4 多 Agent 原生 / Multi-Agent Native

接力（handoff）是一等公民。Agent A 写入的记忆，Agent B 能在新 session 里直接读取使用。换工具不丢上下文。

### 3.5 标准优于实现 / Conformance over Implementation

我们公开**协议规范**和**验收测试套件**——任何第三方都可以实现自己的兼容版本。iHow Memory 不指望靠代码秘密保护商业价值，**指望靠定义标准来沉淀生态**。

### 3.6 安全边界内置 / Built-In Safety Boundaries

记忆系统默认拒绝写入访问凭证、密码、会话字符串、鉴权令牌等敏感内容。每次写入前都跑一遍正则扫描，触发告警直接阻断。

## 4. 5 个核心可靠性场景 / 5 Core Reliability Scenarios

iHow Memory 的可靠性不是抽象承诺，而是用一套**公开验收场景**度量。每个场景定义了"什么叫记忆系统在这种情况下没失忆"。详细规范请见配套文档 `reliability-scenarios-v0.1.md`。

### 场景一：跨工具接力 / Cross-Tool Handoff

Agent A 在某个项目里完成了部分工作并写入记忆。Agent B 在新 session 中接手同一个项目，**应当能直接读取并基于 A 的工作继续推进，不需要用户重新讲一遍**。

### 场景二：反馈规律自动沉淀 / Feedback Pattern Accumulation

用户或客户对同一类输出连续多次给出同向反馈（例如连续 3 次要求"开头别太硬广"）。系统**应当自动识别这是稳定模式而不是偶发偏好**，沉淀为可复用规则，在下一次同类生成中自动应用。

### 场景三：禁忌约束执行 / Constraint Enforcement

项目定义了一组禁忌（如不能提竞品、不能用绝对化用词、不能宣称医学功效）。系统**应当在每次生成前自动应用这些约束**，跨工具跨 session 一致生效，违规情况可被审计追溯。

### 场景四：新人接手 / Team Handoff

一个人接手另一个人（或另一个 AI）正在做的客户项目。新人**应当能从记忆库直接看到当前状态、关键决策、客户禁区、阻塞和下一步**，不需要翻阅过去几个月的聊天历史。

### 场景五：跨模型迁移 / Cross-Model Migration

用户从一个 LLM 切换到另一个 LLM。**同一份本地记忆库应当能被新模型读取使用**，迁移成本接近零。

> **每个场景都附带：触发条件、系统应执行动作、期望结果、失败模式、验收标准**。完整规范在配套文档中。

## 5. Protocol 草案 / Protocol Sketch

iHow Memory 的协议层由 **4 个核心接口**组成。详细 JSON schema 与字段定义请见配套文档 `protocol-draft-v0.1.md`。

### 5.1 `events` — 业务事件写入

业务系统在关键时点（新建项目、客户反馈、人工改稿、决策记录等）向 iHow Memory 推送结构化事件。事件被异步处理后，符合条件的部分进入记忆层。

**核心承诺**：写入幂等、可审计、可回滚。

### 5.2 `context` — 上下文包读取

AI 工具在生成内容前向 iHow Memory 请求上下文包（context pack）。返回的内容**带 source_refs**（每条信息可追溯到来源文件与行号）和**confidence**（可信度标记）。

**核心承诺**：返回的上下文是结构化的、可信度透明的、可被生成模块直接 inject。

### 5.3 `writeback` — 反馈与改稿写回

人工对 AI 输出做出修改 / 反馈时，业务系统调用 writeback。系统**不会立即覆盖长期记忆**——而是进入 pending update 队列，按"低风险自动写 / 中等候选保留 / 高影响人审"三层处理。

**核心承诺**：人不被 AI 反向控制；高影响变更必须显式确认。

### 5.4 `audit` — 审计追溯

任意时刻可以查询记忆是何时写入、由哪个 agent / 哪个事件触发、来源文件路径、写入前后的差异。

**核心承诺**：100% 文件级可追溯、可回滚到任一历史快照。

### 5.5 隔离与边界

- **多租户隔离**：`tenant_id / customer_id / project_id` 三层强制隔离，跨边界访问直接拒绝
- **安全 scrub**：所有写入路径前置敏感词正则扫描，命中即阻断
- **本地默认**：除非显式启用同步，所有数据留在本地

## 6. 验证证据 / Validation Evidence

iHow Memory 的可靠性不是 PPT 承诺，而是用国际公开评测基准量化证明的。

### 6.1 LongMemEval Public-S 完整通过

**LongMemEval** 是当前业内广泛使用的长期记忆能力评测基准，由学术机构发布，多个主流 AI 记忆系统（Mem0 / Zep / Letta 等）都基于它做能力验证。

iHow Memory 在 **Public-S 全部 470 个样本** 上完成测试，结果：

| 指标 | 我们 | 业界公开基线（参考） |
|---|---|---|
| recall_all@10（完整找回率） | **1.000** | ~0.40（无记忆 baseline）/ ~0.65（RAG 增强）|
| recall_any@10 | **1.000** | — |
| NDCG@10（排序质量） | **0.9457** | — |

### 6.2 回归测试常绿

- **M48 / S94 回归测试集**：长期保持 `failed = 0`
- **每次系统更新都跑完整 gate**：可复核可追溯
- **公开 leaderboard 提交**：v0.1 阶段同步完成

### 6.3 多 Agent 实测共用

iHow Memory 已在三个独立 AI 工具之间完成真实跨工具接力测试：
- 一个 agent 在记忆中沉淀决策与状态
- 第二个 agent 在新 session 中读取并接续工作
- 第三个 agent 完成最终交付物
- 全程**无信息丢失、无重复请求用户上下文**

### 6.4 工具中立性验证

iHow Memory 的协议草案以工具中立为目标：不同 AI 工具可以围绕同一份项目记忆语义完成接力，而不依赖某个模型或某个会话的隐藏状态。

v0.1 阶段已经完成内部跨工具验证：

- 不同 agent 可以读取同一份项目状态摘要
- 决策、约束、阻塞和下一步不会被锁在单一会话中
- 接收方可在有限上下文内继续工作，而不是重读完整历史

这些验证用于支撑协议语义，不在 v0.1 公开任何工具侧实现代码。

## 7. Open Standard 倡议 / Open Standard Invitation

记忆系统不应该是某家公司的封闭产品。它应该像 HTTP、SQL、Kubernetes 一样——**一套公开协议 + 一套公开测试 + 多个实现共同进化**。

### 7.1 我们开放什么

| 资产 | 公开级别 |
|---|---|
| 协议规范（spec） | 完全开放，欢迎社区贡献 |
| 5 个核心可靠性场景定义 | 完全开放 |
| 一致性测试的文字规格（conformance specification） | 完全开放 |
| 白皮书与公开说明文档 | 完全开放 |

v0.1 是一个 **Open Standard draft**，不是代码发布。公开仓库不包含 adapter、SDK、runner、server、deploy 配置或参考服务。

### 7.2 我们邀请什么

- **更多 AI 工具的标准讨论**：希望更多 LLM 厂商、AI 编辑器、Agent 框架参与协议语义与互操作边界讨论
- **更多行业场景的扩展**：金融、医疗、法律、教育、内容生产等场景的可靠性场景沉淀
- **更多 conformance 规格贡献**：让"什么叫合格的 AI 记忆系统"成为一个**社区共建的标准**，而非任何单一厂商说了算
- **更多术语、边界和安全审查**：帮助协议在公开讨论中变得更清晰、更可验证

### 7.3 为什么开放

我们相信记忆基础设施有**网络效应**：当越多工具讲同一份协议，用户的迁移成本越低，整个生态越繁荣，受益的是所有参与者。

iHow Memory 团队的商业模式建立在**服务、企业部署、行业模板**之上，而非协议层封闭。我们没有理由把协议关起来，反而有强动力把它做成公认的标准。
