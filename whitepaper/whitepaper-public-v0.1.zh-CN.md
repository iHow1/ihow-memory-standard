# iHow Memory · 白皮书 v0.1

**多 Agent 工作流的本地优先记忆与交接可靠性层**
*Local-First Memory & Handoff Reliability Layer for Multi-Agent AI Workflows*

---

## 摘要 / Executive Summary

AI Agent 在单次对话里已经足够聪明，但跨会话、跨工具、跨人接手之后会**集体失忆**。客户偏好、改稿规律、关键决策、交接边界——这些项目级状态被困在某个聊天历史里，下一次工作必须从头重新讲一遍。

**iHow Memory 解决这个问题**。它是一套面向多 Agent 工作流的**本地优先记忆与交接可靠性层**：用文件协议沉淀项目状态、用通用接口对接任意 AI 工具、用可审计的方式管理人审与回滚、用一套公开验收套件证明系统真的"不失忆"。

iHow Memory 已在团队内部完成多轮长期记忆与多 agent 接力工作流的验证测试，并已在多个 AI 工具之间完成跨工具接力闭环。具体可复核的 conformance 工具将在标准边界稳定后再考虑发布。

本白皮书阐述 iHow Memory 的问题陈述、设计原则、5 个核心可靠性场景、协议草案、验证方向，以及面向社区征求审阅的开放规范草案。

## 1. 问题陈述 / The Problem

### 1.1 现状：AI 在单点聪明，在长链失忆

过去两年 AI Agent 工具爆发——编程助手、内容生成、客户支持、研究分析全部 AI 化。当一个任务在**单次对话**里完成时，AI 表现可以接近专家。

但**真实工作不是单次对话**。一个客户项目持续数周，一段开发任务横跨数月，一份调研要在多个工具之间接力。一旦工作流跨越 session、跨越工具、跨越人，AI 的"聪明"就退化成"健忘"——它每次都从零开始认识你。

### 1.2 三种典型失忆

| 失忆类型 | 真实代价 |
|---|---|
| **改稿失忆 / Revision amnesia** | 客户反馈过 3 次的偏好，AI 第 4 次还会犯同样错误 |
| **工具失忆 / Tool-swap amnesia** | 从一个 AI 工具切到另一个 AI 工具，项目状态、技术决策、风险记录全部清零 |
| **接手失忆 / Handoff amnesia** | 新人接手客户项目，要从聊天记录里翻找过去半年的上下文，效率从第一天起就是 -70% |

### 1.3 现有方案为什么不够

市面上主要的记忆方案，按局限分类：

- **向量数据库黑盒**：能"想起"事实，但人不可读、不可审、不可质疑。一旦出错没办法回滚到具体源头。
- **云端 SaaS 记忆**：方便但要把客户敏感资料上传给第三方。金融、医疗、法律、政企不可行。
- **单 Agent 记忆模块**：每个 AI 工具自带一套记忆，互相不互通。换工具等于重新建模。
- **手工文档 + 知识库**：完全依赖个人维护，长期不可扩展。

### 1.4 真正的瓶颈是"记忆基础设施"

当企业 / 团队 / 个人**从单 Agent 试验进入多 Agent 流程**，记忆问题从"feature"升级到"control plane"——它直接决定 AI 自动化能不能从"一次性脚本"变成"可持续运营资产"。

iHow Memory 就是为解决这个 control plane 问题而设计。

## 2. iHow Memory 定位 / Positioning

### 2.1 一句话定义

> **iHow Memory 是面向多 Agent 工作流的本地优先记忆与交接可靠性层。**
>
> *iHow Memory is the local-first memory and handoff reliability layer for multi-agent AI workflows.*

### 2.2 它不是什么

- 它**不是**通用对话记忆（如 Mem0 风格的"AI 记住用户喜好"）
- 它**不是**向量检索数据库的替代品
- 它**不是**单 Agent 框架（如 Letta / MemGPT 风格）
- 它**不是**云 SaaS 服务

### 2.3 它是什么

- 它是项目级、文件协议、可审计、跨 AI 工具的记忆中台
- 它是多 Agent 之间稳定接力的可靠性层
- 它是企业可以私有化部署、不需要让数据离开本地的基础设施
- 它是公开的协议规范 + 公开的验收测试，**邀请社区共同定义"什么叫合格的 AI 记忆"**

## 3. 核心设计原则 / Core Design Principles

### 3.1 本地优先 / Local-First

记忆数据默认存储在用户或客户自己的环境里。云端是可选项，不是默认值。这让 iHow Memory 天然适合高敏行业。

### 3.2 文件协议 / File Protocol

核心记忆使用 Markdown + YAML frontmatter 存储。人可以直接打开、阅读、修改、回滚。**不是向量黑盒，是可审计的真相源**。

### 3.3 模型无关 / Model-Agnostic

iHow Memory 不绑定任何 LLM。无论后端是主流商业模型、国内模型，还是自部署的开源模型，记忆层的接口一致。

### 3.4 多 Agent 原生 / Multi-Agent Native

接力（handoff）是一等公民。Agent A 写入的记忆，Agent B 能在新 session 里直接读取使用。换工具不丢上下文。

### 3.5 标准优于实现 / Conformance over Implementation

我们公开**协议规范**和**验收方向**——任何第三方都可以审阅、批评并提出自己的兼容思路。iHow Memory 不指望靠代码秘密保护商业价值，而是先把可靠性语言放到社区面前接受审阅。

### 3.6 安全边界内置 / Built-In Safety Boundaries

记忆系统默认拒绝写入访问凭证、密码、会话字符串、鉴权令牌等敏感内容。每次写入前都跑一遍正则扫描，触发告警直接阻断。

## 4. 5 个核心可靠性场景 / 5 Core Reliability Scenarios

iHow Memory 的可靠性不是抽象承诺，而是用一套**公开验收场景**度量。每个场景定义了"什么叫记忆系统在这种情况下没失忆"。详细规范请见配套文档 `reliability-scenarios-v0.1.md`。

1. **跨工具接力 / Cross-Tool Handoff**：已完成工作、阻塞、约束和下一步能在换工具后继续保留。
2. **反馈规律沉淀 / Feedback Pattern Capture**：重复反馈只有在证据足够时才沉淀成有范围的记忆。
3. **禁忌约束执行 / Constraint Preservation**：硬约束会在相关生成或行动前被读取。
4. **新人接手 / Human Team Handoff**：新人可以理解项目状态，而不必重读完整原始历史。
5. **跨模型迁移 / Model Migration**：项目记忆语义能在不同 model-facing clients 之间保持稳定。

每个场景都包含 Given、When、Then、失败模式和验收标准。

## 5. Protocol 草案 / Protocol Sketch

iHow Memory 的协议层由 **4 个核心接口**组成。详细 JSON schema 与字段定义请见配套文档 `protocol-draft-v0.1.md`。

### 5.1 `events`

记录工作流事实：用户指令、决策、文件变更、评审意见、验证结果、交接和阻塞。

### 5.2 `context`

为当前任务读取最小有用上下文包，并尊重范围边界和来源元数据。

### 5.3 `writeback`

把已发生的工作转化为候选长期记忆。根据策略，写回可以先进入人工审核再生效。

### 5.4 `audit`

解释记忆来自哪里、何时变化、为什么被读取、如何被使用。

## 6. 验证方向 / Validation Direction

iHow Memory 的可靠性是用**行为级验收标准**度量的，而不是单一指标。

iHow Memory 已在团队内部进行过长期记忆与多 agent 接力工作流的验证测试，包括 LongMemEval 风格的检索测试与多 agent 续接实验。这一公开版本首先发布**可靠性语言**（reliability language）；可复核、可执行的 conformance 工具将在标准边界稳定后再考虑发布。

公开材料的核心是**行为级验证原则**：

- 系统能否在跨工具之间保留项目状态？
- 系统能否区分硬约束与软偏好？
- 系统能否解释为什么调用了某条记忆？
- 系统能否防止无关项目泄露？
- 系统能否支持审核、回滚与删除？

这五个问题对应 §4 中的五个可靠性场景，是判断一个记忆系统"是否真的不失忆"的可验证标准。

## 7. 开放规范草案邀请 / Open Spec Draft Invitation

记忆系统不应该是某家公司的封闭产品。它应该像 HTTP、SQL、Kubernetes 一样——**一套公开协议 + 一套公开测试 + 多个实现共同进化**。

### 7.1 我们开放什么

| 资产 | 公开级别 |
|---|---|
| 协议规范（spec） | 完全开放，欢迎社区贡献 |
| 5 个核心可靠性场景定义 | 完全开放 |
| 一致性测试的文字规格（conformance specification） | 完全开放 |
| 白皮书与公开说明文档 | 完全开放 |

v0.1 是一个 **open spec draft**，不是代码发布。公开仓库不包含 adapter、SDK、runner、server、deploy 配置或参考服务。

### 7.2 我们邀请什么

- **更多 AI 工具的标准讨论**：希望更多 LLM 厂商、AI 编辑器、Agent 框架参与协议语义与互操作边界讨论
- **更多行业场景的扩展**：金融、医疗、法律、教育、内容生产等场景的可靠性场景沉淀
- **更多 conformance 规格贡献**：让"什么叫合格的 AI 记忆系统"成为一个**社区共建的标准**，而非任何单一厂商说了算
- **更多术语、边界和安全审查**：帮助协议在公开讨论中变得更清晰、更可验证

### 7.3 为什么开放

我们相信记忆基础设施有**网络效应**：当越多工具讲同一份协议，用户的迁移成本越低，整个生态越繁荣，受益的是所有参与者。

iHow Memory 团队的商业模式建立在**服务、企业部署、行业模板**之上，而非协议层封闭。我们没有理由把协议关起来，反而有强动力把它做成公认的标准。

## 9. 结语 / Closing
AI 工作应该是连续的。
用户不应该因为换工具、换模型、换聊天窗口或换人接手，就反复重建同一份项目上下文。iHow Memory 从一个很窄的公开承诺开始：先定义可靠 agent 记忆的共享语言，再讨论实现细节。
